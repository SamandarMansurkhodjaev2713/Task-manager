# UX style guide — тексты бота

Якорный документ для всех видимых пользователю строк (`src/presentation/telegram/ui_text_*.rs`).
Любой PR, добавляющий или меняющий пользовательский текст, должен пройти по этому чеклисту.

---

## 1. Тон и обращение

- **«Вы»** — строчное (нейтрально, без помпезности). Заглавное «Вы» — только в исключительных
  случаях типа официальных писем; в обычных кнопках/подсказках бота — лишнее.
- **Стиль** — деловой, краткий, без канцелярита. Глаголы вместо отглагольных существительных:
  - ✅ «назначить», «открыть карточку», «сообщить о блокере»
  - ❌ «осуществить назначение», «произвести открытие», «выполнить отчёт»
- **Никаких смайликов в кнопках для выражения эмоций** — только функциональные эмодзи
  (см. ниже).

## 2. Терминология (фиксированный словарь)

| Используем | Не используем |
|------------|---------------|
| исполнитель | ответственный, назначенный, ассайни |
| срок | дедлайн (в видимом UI; в логах допустимо) |
| задача | задание, таск, тикет |
| карточка задачи | детали, страница, форма |
| администратор | админ (в видимом UI; служебно — допустимо) |
| менеджер | руководитель, начальник |
| сотрудник | работник, юзер, член команды |
| уведомление | нотификация, нотиф |
| справочник сотрудников | база, БД, директория |
| @username | ник, никнейм, тег |
| код задачи | id, идентификатор, ID |

## 3. Эмодзи

Используем только функциональные эмодзи как **визуальные иконки разделов и статусов**.
Никогда не разбрасываем по тексту для «оживления».

| Эмодзи | Семантика |
|--------|-----------|
| 🏠 | Главное меню |
| 🆕 | Создание нового объекта |
| ⚡ | Быстрый режим |
| 🧭 | Пошаговый режим / Менеджер |
| 📋 | Карточка задачи / задачи |
| 📥 | Входящие задачи |
| 📤 | Созданные / исходящие |
| 📊 | Личная статистика |
| 📈 | Командная статистика |
| ⚙️ | Настройки / профиль |
| ❓ | Справка / помощь |
| 🛡 | Администратор |
| 🔔 | Уведомления |
| 🎤 | Голосовое создание |
| 🔄 | Синхронизация |
| ℹ️ | Информационное сообщение |
| ⚠️ | Предупреждение / опасное действие (нейтральное) |
| ⛔ | Запрет / отмена |
| ✅ | Успех / подтверждение |
| ❌ | Отказ / отмена-кнопка |
| ↩️ | Возврат назад |
| ✏️ | Редактирование |
| 💬 | Комментарий |
| 🚧 | Блокер |
| 🧪 | На проверке / inbox менеджера |
| ▶️ | В работе / взять в работу |
| 📨 | Отправлено / доставлено |
| 👤 | Сотрудник / переназначение |
| 👥 | Группа сотрудников |
| 🚩 | Флаги функций |
| 📜 | Журнал действий |
| 🔐 | Журнал безопасности |
| ⏰ | Время / срок |
| 📅 | Календарь / напоминание |
| 👋 | Приветствие / подключение |
| ✨ | Welcome / завершение онбординга |

**Правила:**
- Один эмодзи на заголовок строки максимум.
- Эмодзи **в начале** строки (не в середине, не в конце).
- На кнопке: эмодзи + один-два слова текста.
- В заголовке экрана: эмодзи + название раздела.

## 4. Пунктуация

- **Длинное тире** «—» (U+2014) с пробелами вокруг: `Фраза — пояснение`.
- **Короткое тире** «–» (U+2013) — только для диапазонов: `22:00 – 08:00`.
- **Кавычки** — «ёлочки» внешние, „лапки" внутренние:
  - ✅ `Кнопка «Создать задачу»`
  - ✅ `Сообщение «Открой, пожалуйста, "task bot" и нажми /start»`
- **Точка в конце:** в завершённых предложениях — да; на кнопках и заголовках — нет.
- **Двоеточие перед списком** — обязательно.
- **Запятые в перечислениях** — Оксфордская запятая не используется (русская норма).

## 5. Структура экрана

```
[ЭМОДЗИ] Заголовок раздела

Один-два предложения контекста — что это за экран, что от пользователя.

Список / карточка / форма.

Если применимо: подсказка о следующем шаге.
```

## 6. Кнопки

- **Длина** — до 30 символов, эмодзи считается за 2.
- **Глагольная форма** — «Создать», «Отменить», «Открыть».
- **Не дублировать смысл** — если кнопка единственная, она не должна повторять заголовок.
- **Опасные действия** — отдельной строкой, с предупреждающим эмодзи (⛔/⚠️).
- **«🏠 В меню»** — последняя строка любого экрана, кроме самого главного меню.
- **«↩️ К <списку>»** — выше «🏠 В меню» при глубокой навигации.

## 7. Сообщения об ошибках

Каждое сообщение об ошибке должно отвечать на 3 вопроса:

1. **Что не получилось** — конкретно, не «произошла ошибка».
2. **Почему** — если можно объяснить за 1 строку.
3. **Что делать** — повторить, подождать, обратиться к администратору.

Примеры:

- ❌ «Произошла ошибка»
- ✅ «Не получилось обновить статус: задача уже изменена другим участником. Откройте карточку заново и попробуйте ещё раз.»

## 8. Запрещено

- ❌ Markdown / HTML без `parse_mode` — Telegram сломается на спецсимволах от пользователя.
- ❌ Длинные строки без переносов: разбиваем на абзацы по 3-7 строк.
- ❌ Раскрытие технических деталей — никаких `error_code: TASK_NOT_FOUND` в видимых строках
  (только в логах).
- ❌ Личные данные в примерах — используем «Иванов», «Петров», `@ivanov` и т.п.
- ❌ Англицизмы где есть русский эквивалент.
- ❌ ЗАГЛАВНЫМИ КАК АКЦЕНТ — Telegram это не любит, лучше эмодзи.

## 9. Чеклист перед коммитом

Любой PR с пользовательским текстом проверяется по списку:

- [ ] Терминология из раздела 2.
- [ ] Тон и обращение из раздела 1.
- [ ] Эмодзи из раздела 3 (если используются).
- [ ] Пунктуация: тире, кавычки, точки в концах предложений.
- [ ] Структура из раздела 5 (заголовок, контекст, тело).
- [ ] Сообщения об ошибках по схеме из раздела 7.
- [ ] Длина каждого экрана < 3800 байт (запас перед лимитом Telegram 4096).
- [ ] Нет Markdown / HTML спецсимволов в шаблонах с пользовательским контентом.
- [ ] Snapshot-тесты на ключевые экраны обновлены или добавлены.
